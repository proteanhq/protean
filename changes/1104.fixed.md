Fixed two Query Handler read-side defects. `@handle` on a `BaseQueryHandler` method now raises `IncorrectUsageError` at `domain.init()` naming `@read` as the correct decorator — previously it registered silently and ran the method inside a `UnitOfWork` with version/transient retry, violating the stateless-read contract. `ReadView.count()` for database-backed projections now issues a single flat `SELECT COUNT(*)` (`self.query.count()`) instead of fetching up to `limit` rows via `.all()` and reading `.total`, which materialized full entities and registered them in the UoW identity map just to report a count. As a side effect, an Elasticsearch-backed projection's `count()` now returns the exact document count instead of the (potentially capped at 10,000) `hits.total.value` reported by `.all()`.

**Breaking-change classification (ADR-0004):** Tier 2 (behavioral) — a query handler with `@handle` previously registered and ran; it now fails at `domain.init()`. No opt-in/warn/flip transition is applied under the *Silent Correctness Bug* exception: the prior behaviour ran reads inside a UnitOfWork against the documented stateless-read contract, so a transition window would only perpetuate the bug. The failure is loud and immediate and the migration is mechanical (`@handle` → `@read`), with no opt-out because a stateless read must never run in a UnitOfWork.
