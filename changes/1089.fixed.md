Fix `protean server --workers N` silently double-processing events on the event-store subscription path. Event-store subscriptions read directly from the event store with no cluster-wide ownership, so every worker processes the same events. The server now refuses to start more than one worker when any handler resolves to an event-store subscription, naming the offending handlers and pointing to the three ways forward: run a single worker, switch those handlers to stream subscriptions (`subscription_type = "stream"`), or pass the new `--allow-event-store-multiworker` flag to override. The guard is enforced at both the `protean server` CLI and the programmatic `Supervisor` entry point (which gains an `acknowledge_event_store_risk` parameter). Stream-based deployments are unaffected and continue to scale horizontally, as do broker-only deployments backed by a cluster-aware broker (e.g. Redis Streams consumer groups). The guard is **per-process**: it only sees the workers within one `protean server` invocation and cannot detect a second process (another container or Kubernetes replica) reading the same event store, so a domain with any event-store subscription must run as exactly one process cluster-wide until database-backed cluster ownership is available. The production-deployment guide documents this boundary.

**Breaking-change classification (ADR-0004):** Tier 2 (behavioral) — a domain with event-store subscriptions and `--workers > 1` previously started (and silently double-processed events); it now refuses to start. No 3-version opt-in/warn/flip transition is applied because the prior behavior was a silent correctness bug, not an intentional default this policy protects; the `--allow-event-store-multiworker` flag / `acknowledge_event_store_risk=True` parameter is the immediate, same-release opt-out for anyone who needs the old behavior.
