Fixed an embedded `ValueObject` whose fields hold default/falsy values (e.g. all zeros) being reconstructed as `None` after persist + reload. The `ValueObject` field gated presence on truthiness — and such a VO is falsy — so it was reset to `None` on assignment and dropped from serialization, losing the distinction between "set to all-defaults" and "never set." Presence is now decided by identity (`is not None`), so an explicitly-assigned VO round-trips to an equal VO across the in-memory, relational, and Message-DB adapters (and on event-sourced aggregates). An unset VO still reads back as `None`.

Two boundaries to note. `bool(vo)` still reports an all-default VO as falsy (unchanged) — check presence with `agg.vo_field is not None`, not `if agg.vo_field:`. And a VO whose *every* field is `None` still reads back as `None` from flattened (relational/document) storage, because all-NULL columns are indistinguishable from an unset VO; give at least one field a non-null default (e.g. `0`) if you need to tell "set" from "unset."
