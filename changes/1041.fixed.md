Make `Outbox.target_broker` NOT NULL so a NULL row can no longer bypass the `(message_id, target_broker)` unique index — the dual-write idempotency guard (NULLs compare as distinct in a unique index, silently reopening the duplicate-publish window). It now defaults to the internal broker name; the framework always sets it on write, and legacy rows with a NULL `target_broker` are coerced to the default broker name on read.

**Schema change (Upgrade Notes):** new deployments get a `NOT NULL target_broker` column automatically. Existing deployments keep their current (nullable) column — Protean does not auto-alter tables — and continue to read any legacy NULL rows safely. To enforce the constraint on an existing database, backfill and add it manually, e.g. `UPDATE outbox SET target_broker = 'default' WHERE target_broker IS NULL;` followed by an `ALTER TABLE ... ALTER COLUMN target_broker SET NOT NULL`.
