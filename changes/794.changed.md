Surface connection pool configuration in domain.toml with production-sensible defaults. SQLAlchemy pool defaults raised to pool_size=5 and max_overflow=10 for PostgreSQL and MSSQL (previously pool_size=2, max_overflow=5). Redis broker, Redis cache, and MessageDB event store now forward pool parameters (max_connections, socket_timeout, etc.) from conn_info. A LOW_POOL_SIZE warning is emitted during domain validation when pool_size is set below 5.

**Breaking change — Tier 2 (behavioural).** A deployment that previously relied on the default SQLAlchemy pool settings will now open up to `workers × 15` connections per domain (was `workers × 7`). Before upgrading, verify that the database's `max_connections` setting has headroom; on PostgreSQL the default `max_connections` is 100. To retain the previous behaviour, set `pool_size = 2` and `max_overflow = 5` explicitly on each `[databases.<name>]` block. The framework will emit a `LOW_POOL_SIZE` warning when the value is below 5 — silence it by setting `PROTEAN_ENV=development` or `testing`. See [Harden the Server → Size the pool against your database's ceiling](docs/guides/server/hardening.md#size-the-pool-against-your-databases-ceiling) for sizing guidance.
