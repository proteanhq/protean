`protean check` diagnostics now carry a richer schema: every built-in finding includes a `category` (`handler_completeness`, `aggregate_design`, `versioning`, or `deprecation`), a `rule` with a `rationale` and a `fix`, and a `suggestion`. A new per-element `suppress_checks` decorator option (e.g. `@domain.aggregate(suppress_checks=["AGGREGATE_TOO_LARGE"])`) silences named codes for a single element, and a `[lint.suppressions]` config table (`{CODE = N}`) grandfathers the first `N` findings of a code in a deterministic order so a codebase can adopt a rule without failing on pre-existing violations. Custom lint rules that emit only the minimal keys still work and default to `category = "custom"`.
