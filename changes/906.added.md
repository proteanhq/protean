Handlers can now auto-retry on transient infrastructure failures: configure `@domain.command_handler(retries=3, backoff="exponential")` (also on `@domain.event_handler`) or enable it domain-wide via `server.transient_retry`. Transient exceptions (`ConnectionError`, `TimeoutError`, `SendError` by default; overridable via `retry_exceptions`) are retried with a configurable backoff strategy (`exponential` | `linear` | `fixed`) before the error reaches the subscription retry/DLQ pipeline. This is distinct from the existing version (OCC) retry, which is specific to `ExpectedVersionError`; the two compose with independent attempt budgets. Each attempt runs in a fresh `UnitOfWork`, so a failed attempt rolls back cleanly before the retry. Opt-in and disabled by default — existing behavior is unchanged. Retries increment the `protean.handler.retried` metric.
