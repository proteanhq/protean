`protean check` now flags **`UNINDEXED_FILTER_PATH`**: a repository query that filters on an aggregate field no declared index covers, which forces a full table scan on a relational backend. The rule is the first consumer of the behavioral-analysis substrate — it joins the indexes declared on the aggregate with the fields each repository query names, read from method bodies (`filter`/`get`/`find`/`find_by`/`exclude`) across the whole package, including application-service queries whose receiver statically resolves to a registered aggregate or repository class. A field is covered when it is the identifier, a single-column `unique` constraint, or the leading column of an `Index`; a filter on a non-leading composite column, or on any other scalar field, is flagged with the call-site location and an `Index()` fix. An operator lookup such as `age__gte` is evaluated by its base field. It is deterministic and conservative — a dynamic `filter(**kwargs)`, a value-object or association field, or an unresolvable receiver (a local variable, or a `repository_for(...)` result pending dataflow) is skipped, not guessed — and suppressible per-aggregate via `suppress_checks=["UNINDEXED_FILTER_PATH"]` or the `[lint].suppressions` allow-list. See the [rule reference](concepts/protean-check/rules/unindexed-filter-path.md).
