Add the `UNBOUNDED_INDEXED_STRING` build-time lint rule (category `persistence`, warning level): `protean check` now flags an aggregate field that is both included in a declared index and an unbounded `String` or `Text` — specifically a `Text` field, or a `String` declared with `max_length=None`. Such an index produces DDL that fails on SQL Server, needs an explicit prefix length on MySQL, and carries storage and performance overhead on PostgreSQL. Give the field a bounded `max_length`, or remove it from the index. The rule is scoped to an aggregate root's own scalar fields named in a declared `Index`; other string-like types (`Identifier`, `Status`), value-object attributes, field-level `unique=True`, and child-entity/projection indexes are out of scope. Because it is a warning-level rule, a domain that already declares such an index will newly report a finding (and, at the default `[lint].level = "warn"`, fail `protean check`).
